设计模式之创建类模式笔记

发表于 2017-08-18   |   分类于 设计模式   |   阅读次数

前言

创建类模式包括工厂方法模式、建造者模式、抽象工厂模式、单例模式和原型模式，提供对象的创建和管理职责。

单例模式：保持在内存中只有一个对象。

除了考虑线程并发情况，还要考虑对象复制情况。

实现对象复制，需实现Cloneable接口，并实现了clone方法。对象复制是不用调用类的构造函数就可以复制对象。因此单例类不要实现Cloneable接口。

饿汉式单例.在类初始化时，已经自行实例化

懒汉式单例.在第一次调用的时候实例化自己

假设没有关键字volatile的情况下，两个线程A、B，都是第一次调用该单例方法，线程A先执行instance = new Instance()，该构造方法是一个非原子操作，编译后生成多条字节码指令，由于JAVA的指令重排序，可能会先执行instance的赋值操作，该操作实际只是在内存中开辟一片存储对象的区域后直接返回内存的引用，之后instance便不为空了，但是实际的初始化操作却还没有执行，如果就在此时线程B进入，就会看到一个不为空的但是不完整（没有完成初始化）的Instance对象，所以需要加入volatile关键字，禁止指令重排序优化，从而安全的实现单例。

创建对象可以分解为如下的3行伪代码
memory=allocate(); //1:分配对象的内存空间
ctorInstance(memory); //2:初始化对象
instance=memory; //3:设置instance指向刚分配的内存地址

登记式单例(多例模式)

原型模式：要求通过复制来产生一个对象。

使用场景：
1.初始化需要消耗非常多资源。
2.通过new对象需要繁琐的数据准备和访问权限。
3.在循环体内产生大量对象时使用，可优化性能。
4.一个对象多个修改者。
如电子账单：内容填充到模版里

优点
性能好：原型模式是在堆内存中二进制流的拷贝，比new一个对象性能好的多。
在内存中直接拷贝，执行函数是不会执行的，减少了约束。

浅拷贝

java做了个偷懒的拷贝动作，clone只是拷贝本对象，但还是指向原生对象的内存元素地址。其对象内部数组和引用对象（满足类成员+可变引用）都不拷贝，但原始对象，如int、long等都会被拷贝。String也会被拷贝，因为String通过字符串池在需要的时候才会在内存中创建新的字符串。
注：使用clone方法，类成员变量禁用final关键字。

工厂方法模式

定义：定义一个用于创建对象的接口，让子类决定实例化哪个类。工厂方法使一个类的实例化延迟到其子类。

优点
低耦合： 创建产品类只要知道产品类名或者约束字符串就可以了，不用知道创建对象的艰辛，降低模块耦合。
扩展性： 增加或适当修改工厂类就可以拥抱变化。
屏蔽产品类：产品类的实例化是工厂类负责，例如切换数据库。
使用场景

  • 是new一个对象的替代品，慎重考虑是否增加一个工厂类进行管理，增加代码复杂度。
  • 需要灵活可扩展的框架时。如，支持多种协议的连接服务器框架。
  • 异构项目中。等等。

简单工厂模式

如果一个模块仅需要一个工厂类，使用静态即可。

多工厂模式

例如一个产品类有5个具体实现，且每个实现类的初始化方法都不相同，如果都写在createHuman方法里会导致臃肿，此时则可以采用多工厂模式。


优点
创建类的职责清晰，结构简单。
缺点
可扩展和可维护性不好，工厂类与产品类需一一对应，因此维护时需要考虑两个对象间关系。
可加协调类：封装子工厂类，对高层模块提供统一访问接口。

延伸

工厂方法实现单例

可以用一个单例构造器，所有产生单例的类都遵循一定的规则，一个类型获得唯一一个实例。

延迟初始化

抽象工厂模式

定义：为创建一组相关或相互依赖的对象提供一个接口，而且无须指定它们的具体类。

在多个业务品种，业务分类时，采用抽象工厂产生对象是一个很好的解决方式。

AbstractCreator职责：定义每个工厂要实现的功能。

优点
封装性：对象创建由工厂类负责，高层模块只要知道对应工厂类即可。
产品族内约束为非公开状态。
缺点
扩展困难：如增加一个产品C，产品族扩展困难，产品等级扩展容易。
使用场景
一个对象族（或者一组没有任何关系的对象）有相同约束，则可以采用抽象工厂模式，即有交叉点，如

建造者模式（又名生成器模式）

定义：将一个复杂对象的构建与它的表示分离，使得同样的构建过程可以创建不同的表示。

  • Product产品类
    通常实现了模板方法模式
  • Builder抽象建造者
    规范产品组建，一般由子类实现。
  • ConcreteBuilder具体建造者
    实现抽象类定义的所有方法，并返回一个组建好的对象。
  • Director导演类
    负责安排已有模块顺序，然后告诉Builder开始建造。

有几个产品类就有几个具体建造者类，且多个产品类具有相同接口或抽象类。

优点

  • 封装性，不必知道产品内部组成细节。
  • 建造者独立，容易扩展。
  • 便于控制建造过程细节，而不会影响其他模块。

使用场景

  • 相同方法，不同执行顺序，产生不同事件结果时，可以采用建造者模式。
  • 不同零件装配到一个对象中，产生的运行结果又不同时。
  • 产品类非常复杂或者调用顺序不同产生不同效果。
  • 在对象创建过程中会使用到系统中的一些其他对象，而这些对象的创建过程复杂时，可以采用建造者封装该对象创建过程。
